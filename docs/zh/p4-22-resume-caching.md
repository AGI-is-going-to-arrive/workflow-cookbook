# 第 22 章 · 断点续传与缓存

> **改了长流水线的第 8 步，前 7 步昂贵的结果直接秒级复用。这就是断点续传（resume）。用 `{ scriptPath, resumeFromRunId }` 重跑，未改动的 `agent()` 调用命中缓存，只有被编辑的及其之后的才重新真跑。**
>
> 这是进阶篇的收官章，也是让前面那些「昂贵的多 agent 流水线」真正能**反复修改、反复运行**的关键。同时完整解释了一个贯穿全书的禁令：为什么脚本中禁用 `Date.now()` 和 `Math.random()`。

---

## 22.1 痛点：改一步，却要从头再跑一遍

假设你写了一条 8 阶段的深度研究流水线，每阶段并行派发若干 agent，整条跑完要烧 50 万 token、花上几分钟。跑完一看，第 8 步（最终报告的措辞）不太满意，于是你改了那一步的 prompt。

没有续传，唯一的选择是**从头把整条流水线再跑一遍**。前 7 步那 40 多万 token 的计算结果没有任何变化，却需要重新执行、再等几分钟。迭代一条长流水线时代价很高：末尾的措辞每调整一次，就要为全程买一次单。

断点续传用于消除这种浪费。它的承诺是：

> **同样的脚本 + 同样的 args → 100% 缓存命中。** 只有你**动过**的 `agent()` 调用（以及它之后的调用）才会重新真跑；没动的，秒级就把上次的结果给你。

这样，迭代长流水线变成了：修改第 8 步 → 重跑 → 前 7 步瞬间命中缓存 → 只有第 8 步真正执行。耗时从几分钟降到几秒。

<div class="callout info">

**官方语义（据 `_grounding.md` A/B 节）**：`WorkflowInput.resumeFromRunId?: string` 就是断点续传：**未改动的 `agent()` 调用返回缓存结果；仅同会话**。要配合 `scriptPath`（磁盘脚本路径，每次调用都会写入磁盘）一起用。`WorkflowOutput` 里那个 `runId`（形如 `wf_...`），就是续传时你要传给 `resumeFromRunId` 的值。

</div>

## 22.2 机制：脚本即文件 + runId 锚点

要理解续传的工作原理，需要回顾第 01 章的两个事实，它们在这里结合起来：

**事实一：脚本即文件。** 据 `_grounding.md`，每次调用 Workflow，运行时都会把脚本**保存到磁盘**，写成会话目录下的一个 `.js` 文件，并在返回里给出 `scriptPath`。你的工作流不是一段转瞬即逝的字符串，而是一个**摆在磁盘上、随时能改的文件**。

**事实二：每次运行有一个 runId。** 据 `_grounding.md`，`WorkflowOutput` 会返回 `runId`（形如 `wf_...`）。它是这次运行的唯一标识，也是这次运行所有 agent 结果缓存挂靠的**锚点**。

续传就是将这两者结合：

1. 第一次跑：拿到 `runId`（如 `wf_abc123`）和已写入磁盘的 `scriptPath`。运行时把每个 `agent()` 调用的结果，按它在脚本里的「身份」缓存下来，全挂在这个 `runId` 底下。
2. 你 `Edit` 那个 `scriptPath` 文件，动了其中某个 `agent()` 调用。
3. 重跑：传 `{ scriptPath, resumeFromRunId: 'wf_abc123' }`。运行时拿缓存一对照，**没改的 `agent()` 调用直接取缓存结果**，改了的（以及它之后的）才重新执行。

```mermaid
flowchart TD
    R1["首次运行<br/>{ script }"] --> ID["得到 runId = wf_abc123<br/>+ scriptPath 写入磁盘"]
    ID --> CACHE["运行时缓存每个 agent()<br/>结果挂在 wf_abc123 下"]
    CACHE --> EDIT["Edit scriptPath<br/>改动第 8 步"]
    EDIT --> R2["重跑<br/>{ scriptPath, resumeFromRunId: wf_abc123 }"]
    R2 --> DIFF{"对照缓存：<br/>该 agent() 改动了吗？"}
    DIFF -->|未改| HIT["命中缓存<br/>秒级返回旧结果"]
    DIFF -->|已改/在改动点之后| RERUN["重新真跑"]
```

这个迭代循环（修改文件 + `scriptPath` 重跑），正是第 01 章所述「想迭代？直接 Write/Edit 那个文件，再用 `{ scriptPath }` 重新调用，不用重发整段脚本」的完整版。续传在此基础上增加了「用 `resumeFromRunId` 复用缓存」这一关键能力。

<div class="callout warn">

**「仅同会话」是一条硬限制。** 据 `_grounding.md`，断点续传只在**同一个会话**内有效。官方明确指出：**工作流还在运行时如果退出 Claude Code，下次会话会从头重跑这条工作流**（原话 *"the next session starts the workflow fresh"*）。不存在跨会话续传。缓存的生命周期绑定在当前会话上，无法关闭 Claude Code 后用之前的 `runId` 来续传。续传是「**在当前迭代会话内**反复调试一条流水线」的工具，不是「隔天接着执行」的持久化方案。需要跨会话保留的状态，应通过其他方式实现（比如让 agent 把产物写到磁盘文件，参见[第 19 章](#/zh/p4-19)的控制面/数据面思想）。

这里讲的 `resumeFromRunId` 是**脚本侧**的程序化续传；它在终端里还有个对应的正式操作：在 `/workflows` 视图里选中一个停掉的 run 按 `p` 续传，走的是同一套缓存机制。两个面的对照见[《官方操作面板》](#/zh/p2-ops)。

</div>

## 22.3 揭晓禁令：为什么不准用 Date.now() 和 Math.random()

到这里，可以完整解释第 01、02 章反复提及但未展开的那个禁令了。

据 `_grounding.md`「硬约束」：脚本**禁用 `Date.now()` / `Math.random()` / 无参 `new Date()`**。第 01 章给的理由是「它们会破坏可重放性」。这一节说明**为什么续传需要可重放性、以及这两个函数是如何破坏它的**。

续传的整个前提就是「**同样的脚本，必然跑出同样的执行**」。只有这样，运行时才敢判定「这个 `agent()` 调用没变，能用缓存」。而这个判定又押在一个假设上：**脚本的逻辑是确定性的、可重放的**，同样的输入，每次跑到这一步时状态都一样。

而 `Date.now()` 和 `Math.random()` 正好**违背**了这个假设：

- `Date.now()`：每次调用都返回不同的时间戳。如果脚本用它来拼接 prompt（比如 `agent(\`分析 ${Date.now()} 之前的数据\`)`），那**同一个 `agent()` 调用，每次重跑时 prompt 都不同**。调用已经「变化」，缓存判定逻辑随之失效。
- `Math.random()`：每次返回不同的随机数。同理，任何依赖它的 `agent()` 调用都无法重放。

```javascript
// ❌ 错误（示意，未实跑）—— 破坏可重放性，会被拒绝
const ts = Date.now()                              // 禁用
const pick = items[Math.floor(Math.random() * 3)]  // 禁用
await agent(`分析 ${ts} 的 ${pick}`)               // 每次重跑都不同 → 续传失效
```

替代方案在 `_grounding.md` 中有明确说明：

**需要时间戳 → 通过 `args` 传入，或在工作流完成后补充。** 将时间作为参数从外部传入（`args.timestamp`），脚本内部就保持了确定性：同样的 `args`，同样的执行。或者等工作流完成后，在外部为结果添加时间戳。

```javascript
// ✅ 正确（示意，未实跑）—— 时间戳由 args 传入，保持可重放
await agent(`分析 ${args.cutoffDate} 之前的数据`)
```

**需要随机性/多样性 → 用 agent 的下标（index）来区分提示词。** 这正是第 17 章「多验证者投票」中使用的方法：用 `i` 给每个 agent 分配不同的视角，既实现了多样性，又保持完全确定（同样的下标 → 同样的 prompt）。

```javascript
// ✅ 正确（示意，未实跑）—— 用 index 而非 random 制造差异
const views = ['性能', '安全', '可读性']
await parallel(views.map((v, i) => () => agent(`从${views[i]}角度审查…`)))
```

<div class="callout tip">

**记住这条因果链**：续传要省钱 → 续传需要判定「调用没变」→ 判定依赖脚本的可重放性 → 可重放就必须禁止不确定性 → 因此禁用 `Date.now()` / `Math.random()` / 无参 `new Date()`。这个禁令不是运行时的限制，而是**「可迭代的长流水线」的必然代价**。理解这条链之后，自然会主动将所有不确定性移到脚本外部（`args`），或用下标替代。

</div>

## 22.4 实战：迭代一条长流水线

将机制落到实际操作上。假设正在迭代一条研究流水线，流程如下：

**第一步，首次运行，拿到 runId。** 照常启动工作流，从完成通知/返回里把 `runId` 和 `scriptPath` 记下来：

```text
Run ID: wf_abc123
Script file: .../workflows/scripts/research-pipeline-wf_abc123.js
```

**第二步，编辑已写入磁盘的脚本。** 用 `Edit` 工具直接修改 `scriptPath` 指向的文件，比如只改最后一个汇总 agent 的 prompt。**关键：不要修改前面阶段的任何 `agent()` 调用**，否则它们的缓存会全部失效。

**第三步，带着 resumeFromRunId 重跑。** 续传前有个官方前置条件：先用 `TaskStop` 把上一次那个 run 停掉（续传只在同一会话内有效）。停好之后，再调一次 Workflow 工具，这回传：

```javascript
// （示意，未实跑）—— 续传调用的入参形态
{
  scriptPath: '.../research-pipeline-wf_abc123.js',
  resumeFromRunId: 'wf_abc123'
}
```

运行时会将前面所有未修改的阶段全部走缓存，只重跑被修改的 agent 及其下游。前几个阶段**秒级**完成（缓存命中），算力只投入到改动点之后。

```mermaid
sequenceDiagram
    participant You as 你
    participant WF as Workflow 运行时
    You->>WF: 首次 { script }
    WF-->>You: runId=wf_abc123 + scriptPath
    Note over You: Edit 脚本，只改第 8 步
    You->>WF: { scriptPath, resumeFromRunId: wf_abc123 }
    WF-->>WF: 第 1-7 步：缓存命中（秒级）
    WF-->>WF: 第 8 步：检测到改动，重跑
    WF-->>You: 新结果（仅付第 8 步的代价）
```

<div class="callout tip">

**真实运行印证（5 个 agent 的流水线，续传 = 0 token / 3 毫秒）**：本书跑了一条 5-agent 的模型解析工作流（Run `wf_9c94951d-58c`），首跑是真实执行；接着用**完全没动过的脚本** + `{ scriptPath, resumeFromRunId: 'wf_9c94951d-58c' }` 原样再跑一遍。两次用量摆一块对比（同一 Run ID）：

| 运行 | agent 数 | total_tokens | duration_ms |
|---|---|---|---|
| 首次（真实执行） | 5 | **133,691** | **32,959** |
| 续传（100% 缓存命中） | 5（全缓存） | **0** | **3** |

续传返回的 5 个结果与首跑**完全一致**。**5 个 agent 续传时全部命中缓存：0 新 token、3 毫秒返回**（首跑为 13 万 token、33 秒）。运行时直接从 journal 中回放每个 `agent()` 的结果，没有重新派发任何 subagent。这将「同脚本 + 同 args → 100% 命中」从承诺变成了实测数据，同时实证回答了下一节的问题「缓存命中是否计入 token」：**不计入**。原始记录见 `assets/transcripts/api-facts-r4.md`（另有一个更早的单 agent 续传 `wf_dacbd480-d5d`，0 token / 8ms，结论相同，见 `assets/transcripts/advanced.md`）。

</div>

<div class="callout warn">

**改动点之后的所有调用都会重跑，即使它们本身没有任何修改。** 续传的失效范围是「从改动点起向后全部失效」：第 8 步一变，第 9、10 步的输入可能随之改变，因此它们也必须重跑以保证结果正确。**设计建议：把最可能反复调整的步骤放在流水线后面**，这样缓存收益最大。如果经常修改第 2 步，那第 3 步之后全部需要重跑，续传能节省的就很有限。将「稳定且计算密集的」步骤放在前面，「需要反复打磨的」步骤放在后面，这就是面向续传的流水线设计原则。

</div>

<div class="callout info">

**「什么算改动」：哪些字段会让一个 `agent()` 丢掉缓存？** 本书实测到的边界是 **同脚本 + 同 args = 100% 命中**（`wf_9c94951d-58c`），这一条是实打实跑出来的。在这之上，R8 又做了受控实测（基线 `wf_4ffde230-535`，3 个 agent / 91,044 token），把两个字段单独拎出来隔离：**只改某 agent 的 `label`（其余不动）→ 续传 0 token 全命中 => `label` 不入缓存键**；**只改它的 `prompt`（label 还原）→ 91,044 重跑成 60,702 token（约等于基线 2/3），改动点之前的 agent 照样命中、该 agent 及其下游重跑 => `prompt` 入键**。最稳的心智模型还是保守那一套：**只要你动了喂给 agent 的 `prompt`、或它依赖的上游数据，就当它丢了缓存**；想稳稳吃到缓存，就让脚本和 args 一个字都别动（只改你确实想重跑的那一步及其下游）。

至于缓存键里**其余字段到底入不入键**，社区第三方资料声称 `opts` 里的 `schema / model / isolation / agentType` 都入键、`phase` 只用来显示、排在键外。**这几个字段本书尚未逐一隔离验证**，所以整体标成**社区第三方资料声称、本书未独立实测**（已实测的 `label`/`prompt` 见上，不算在内）。实践上你不用去背那张精确清单，记住「改了 prompt/上游数据就当缓存失效」这条保守的粗粒度规则，就能安全迭代。

</div>

## 22.5 续传与 budget、嵌套的相互作用

续传不是一个孤立的特性，它与前面几章的机制存在一些容易忽略的关联，理清这些关联有助于避免问题。

**与 budget（[第 21 章](#/zh/p4-21)）的关系：缓存命中是否计入 token？** 续传的价值在于「命中的调用不重新执行」，既然不执行，自然不消耗模型推理的 token。本书的实测已经证实了这一点：5-agent 流水线缓存命中的那次重跑 `total_tokens=0`（见上面「真实运行印证」，Run `wf_9c94951d-58c`，原始记录 `assets/transcripts/api-facts-r4.md`）。续传确实节省 token。**迭代的边际成本只来自被修改的部分**，前面命中缓存的阶段近乎免费。

**与嵌套 `workflow()`（第 20 章）的关系。** 续传所述的「未改动 `agent()` 命中缓存」，针对的是当前工作流脚本中的 `agent()` 调用。如果脚本中包含 `workflow()` 子调用，续传与子工作流的缓存如何互动，事实源没有展开说明，属于「（待核实）」。在迭代带嵌套的工作流时，需要通过 `/workflows` 观察实际的缓存命中行为来确认。

**和 worktree（第 19 章）的关系。** worktree 隔离出来的 agent，会牵涉文件系统的副作用。续传重跑改动点之后那些 agent 时，这些副作用怎么处理（重新建 worktree？），同样是事实源没覆盖的细节，标「（待核实）」。

<div class="callout info">

**一个稳妥的实践原则**：续传最可靠、官方支持最明确的场景，是「**纯读、纯产出结构化数据**的多阶段流水线」，如研究、审查、分析。这类工作流的 `agent()` 调用没有外部副作用，缓存命中的语义清晰（同样输入 → 同样输出 → 可安全复用）。对于带文件写入（worktree）或包含子工作流的复杂情形，续传的行为有事实源未覆盖的细节，**应通过 `/workflows` 观察实际命中情况**，不要依赖假设。这与全书「严禁凭记忆臆测 API」的原则一致。

</div>

## 22.6 续传友好的设计清单

将本章的经验整理为一份「续传友好」的设计清单：

| 原则 | 做法 | 理由 |
|---|---|---|
| **杜绝不确定性** | 禁 `Date.now()` / `Math.random()` / 无参 `new Date()` | 它们破坏可重放性，续传判定失效（会被拒绝） |
| **不确定性赶到外面** | 时间戳用 `args` 传入；多样性用 `index` | 保持脚本确定性，同输入同执行 |
| **易变步骤靠后放** | 稳定昂贵的在前，反复打磨的在后 | 改动点之后全部重跑，靠后改动缓存收益最大 |
| **善用脚本写入磁盘** | 迭代时 `Edit` 磁盘上的脚本 + `scriptPath` 重跑 | 无需重发整段脚本，且为续传提供文件锚点 |
| **记牢 runId** | 首次运行后记下 `runId` 备续传 | `resumeFromRunId` 的值来源 |
| **同会话内迭代** | 续传只在同会话有效 | 跨会话需另靠磁盘持久化 |
| **复杂情形先观察** | 含 worktree/嵌套时用 `/workflows` 看命中 | 这些场景的续传细节事实源未覆盖 |

<div class="callout tip">

**续传把「写工作流」变成了真正的「编程」体验。** 没有续传，每改一次脚本都得付全程的代价，迭代成本高到你不敢随便动一下，那更像「一次性把批处理作业丢出去」。有了续传，改一行、重跑、秒级就看到局部效果，跟在 REPL 里调代码一个样：**改动很便宜、反馈很即时**。这正是 Workflow「确定性脚本」相比「概率性 prompt 编排」的一大工程优势：确定性让缓存成为可能，缓存又让快速迭代成为可能。

</div>

## 22.7 本章小结

- **断点续传（resume）**：用 `{ scriptPath, resumeFromRunId }` 重跑，**未改动的 `agent()` 调用秒级命中缓存**，只有被编辑的、以及它**之后**的调用才重新真跑。承诺是「同脚本 + 同 args → 100% 命中」。
- 机制 = **脚本即文件**（每次调用写入磁盘 `scriptPath`）+ **runId 锚点**（`WorkflowOutput.runId` 是缓存的挂靠点，也是 `resumeFromRunId` 的值）。
- **续传仅同会话**有效；跨会话保留状态需要通过其他方式实现（如让 agent 写磁盘）。
- 揭晓禁令的因果链：续传省钱 → 得判定「调用没变」→ 得靠脚本**可重放** → 禁不确定性 → 故禁 `Date.now()` / `Math.random()` / 无参 `new Date()`。替代：时间戳用 `args`、多样性用 `index`。
- **续传友好设计**：把爱变的步骤往流水线后面放（改动点之后全部重跑），稳定烧钱的放前面。
- 和 budget/嵌套/worktree 的精细交互，有事实源没覆盖的地方（标「（待核实）」）；最可靠的场景是**纯读、纯产出结构化数据**的多阶段流水线，复杂情形拿 `/workflows` 看实际命中。

这一章为进阶篇画上句号。从对抗验证、循环到干，到 worktree 隔离、嵌套、动态预算、断点续传，Workflow 在生产环境中所需的进阶能力已经全部覆盖。下一部分转向社区：在原生 Workflow 出现之前，四大编排系统是如何「模拟」这些能力的，又有哪些精华值得用 `phase`/`schema` 重写成可复用的工作流。
